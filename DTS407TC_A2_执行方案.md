## 一、项目概述与动机

### 问题定义
用户输入一个辩题和目标方向（正方/反方），系统输出该方向下3条有力的辩论论点。

### 核心动机
传统单次生成论点的方式存在明显缺陷：
- 论点未经对方视角检验，存在逻辑漏洞
- 缺乏对反方可能反驳的预判和应对
- 生成内容浮于表面，缺乏真实数据支撑

本系统提出**对抗式论点生成**机制：让正反两方Agent先内部打一场辩论，迫使论点在对抗中被强化，最终输出的论点已经预先应对了反方的核心攻击角度。

### 现实价值
- 辩论赛选手的论点准备工具
- 商业提案/决策的对抗性预演
- 批判性思维训练辅助系统

## 二、系统架构

### 整体流程

```
用户输入：辩题 + 目标方向（正/反）
              ↓
    ┌─────────────────────────────────┐
    │           并行展示               │
    └──────┬──────────────────────────┘
           │
    ┌──────┴──────┐
    ▼             ▼
 策略一          策略二
单Agent         4Agent对抗辩论
+ Web Search    + Web Search
    │               │
    │         ┌─────▼──────┐
    │         │ 第一轮：立论 │
    │         │ 正A 正B     │
    │         │ 反A 反B     │
    │         └─────┬──────┘
    │               │
    │         ┌─────▼──────┐
    │         │ 第二轮：反驳 │
    │         │ 正A 正B     │
    │         │ 反A 反B     │
    │         └─────┬──────┘
    │               │
    │         ┌─────▼──────┐
    │         │  综合Agent  │
    │         │  提炼论点   │
    │         └─────┬──────┘
    ▼               ▼
  3条论点         3条论点
```

### 策略一：单 Agent

| 属性 | 内容 |
|------|------|
| Agent数量 | 1个 |
| 工具 | Web Search |
| 流程 | 搜索辩题资料 → 直接生成3条论点 |
| LLM调用次数 | 1次 |

### 策略二：4 Agent 对抗 + 综合

#### 辩手角色设计

| Agent | 角色 | 立场 | 专长 |
|-------|------|------|------|
| 正方A | 逻辑型辩手 | 正方 | 价值观推理、逻辑链条构建 |
| 正方B | 数据型辩手 | 正方 | 真实案例、统计数据、现实证据 |
| 反方A | 逻辑型辩手 | 反方 | 攻击正方逻辑漏洞、概念辨析 |
| 反方B | 数据型辩手 | 反方 | 反例举证、数据反驳 |
| 综合Agent | 裁判+提炼者 | 中立 | 阅读8条发言，提炼目标方向最强3条论点 |

#### 完整对话流程（9次LLM调用）

**第一轮：立论（4次）**
```
正方A → Web Search → 从逻辑/价值角度提出支持论点
正方B → Web Search → 用数据/案例提出支持论点
反方A → Web Search → 从逻辑角度提出反对论点
反方B → Web Search → 用数据/案例提出反对论点
```

**第二轮：反驳（4次）**
```
正方A → 读取反方立论 → 针对性反驳 + 强化自身论点
正方B → 读取反方立论 → 针对性反驳 + 强化自身论点
反方A → 读取正方立论 → 针对性反驳 + 强化自身论点
反方B → 读取正方立论 → 针对性反驳 + 强化自身论点
```

**综合阶段（1次）**
```
综合Agent → 读取全部8条发言 → 提炼用户指定方向的3条最强论点
```

## 三、技术栈

| 模块 | 技术选择 | 说明 |
|------|---------|------|
| **LLM** | Qwen3.5-4B（Ollama 本地部署）| 开源模型，无需 API Key，本地运行无费用，符合开源要求 |
| **Web Search** | 博查 API（Bocha）| 中文内容检索质量高，有免费 tier |
| **后端** | Python + FastAPI | 轻量，支持流式输出 SSE |
| **前端** | React + Tailwind CSS | 实时显示Agent发言过程 |
| **Agent 框架** | 纯 Python 状态机 | 逻辑简单，不需要 LangGraph |

### 本地模型部署
使用 Ollama 在本地运行 Qwen3.5-4B：
```bash
# 安装 Ollama 后执行
ollama pull qwen3.5:4b
ollama serve  # 默认监听 http://localhost:11434
```
代码中调用方式（兼容 OpenAI SDK）：
```python
from openai import OpenAI
client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
response = client.chat.completions.create(model="qwen3.5:4b", messages=[...])
```

### Thinking 模式设计（重要设计点）
Qwen3.5 支持 **thinking / non-thinking 两种推理模式**，Small 系列（含 4B）默认关闭 thinking。

本项目利用这一特性在两种策略间制造额外差异：

| 策略 | 模式 | 理由 |
|------|------|------|
| 策略一（单Agent）| `/no_think`（默认）| 追求快速直接输出，代表"直觉式"论点生成 |
| 策略二（4辩手+综合）| `/think` | 开启深度推理，代表"思考式"对抗论点生成 |

在每个 Agent 的 prompt 首行加入 `/think` 或 `/no_think` 即可切换模式。这一差异可在报告的实验设计章节中作为额外对比变量加以说明。

## 四、Agent System Prompt 设计

### 策略一：单 Agent Prompt

```
/no_think
你是一位经验丰富的专业辩手。

用户将给你一个辩题，以及需要你支持的方向（正方或反方）。

你需要：
1. 首先使用搜索工具，搜索该辩题的相关背景、数据和案例
2. 基于搜索结果，输出该方向下3条有力的辩论论点
3. 每条论点需包含：核心主张 + 支撑逻辑 + 具体证据

输出格式：
论点一：[标题]
- 核心主张：...
- 逻辑支撑：...
- 具体证据：...

（论点二、三同上）
```

### 策略二：各 Agent Prompt

**正方A（逻辑型）**
```
/think
你是一位擅长逻辑推理和价值论证的正方辩手。

辩题：{topic}
你的立场：正方

请从逻辑推理、价值判断、概念界定的角度，
先搜索相关资料，然后提出2-3条支持正方的核心论点。
注重论证的逻辑严密性，而非堆砌数据。
```

**正方B（数据型）**
```
/think
你是一位擅长运用数据和案例的正方辩手。

辩题：{topic}
你的立场：正方

请搜索真实的统计数据、研究报告、成功案例，
提出2-3条以事实为基础的正方论点。
每条论点必须有具体数据或案例支撑。
```

**反方A（逻辑型）**
```
/think
你是一位擅长逻辑分析和概念辨析的反方辩手。

辩题：{topic}
你的立场：反方

请从逻辑漏洞、概念混淆、价值冲突的角度，
先搜索相关资料，然后提出2-3条反对正方的核心论点。
重点找出正方立场在逻辑上的根本缺陷。
```

**反方B（数据型）**
```
/think
你是一位擅长用反例和数据反驳的反方辩手。

辩题：{topic}
你的立场：反方

请搜索能够反驳正方观点的真实数据、反例和研究，
提出2-3条以事实为基础的反方论点。
```

**反驳轮附加指令（在原Prompt基础上追加）**
```
以下是对方的立论内容：
{opposing_arguments}

请在你原有论点的基础上：
1. 针对对方最薄弱的1-2个论点进行反驳
2. 根据对方的攻击，强化和修正你自己的论点
```

**综合Agent**
```
/think
你是一位资深辩论裁判和论点分析师。

以下是关于辩题「{topic}」的完整辩论记录：

正方A（逻辑型）立论：{pro_a_round1}
正方B（数据型）立论：{pro_b_round1}
反方A（逻辑型）立论：{con_a_round1}
反方B（数据型）立论：{con_b_round1}

正方A 反驳：{pro_a_round2}
正方B 反驳：{pro_b_round2}
反方A 反驳：{con_a_round2}
反方B 反驳：{con_b_round2}

请站在{target_side}的立场，从以上辩论中提炼出3条最有力的论点。
要求：
- 论点要能够应对对方的主要反驳
- 综合逻辑论证与数据支撑
- 每条论点简洁有力，适合实际辩论使用

输出格式：
论点一：[标题]
- 核心主张：...
- 逻辑支撑：...
- 对方反驳应对：...

（论点二、三同上）
```

- 覆盖不同复杂度（用于评估系统稳健性）

## 七、评估方案

### 定量指标

| 指标 | 计算方法 | 预期结论 |
|------|---------|---------|
| **论点多样性** | 3条论点两两 embedding 余弦相似度均值，越低越多样 | 策略二更多样 |
| **LLM-as-Judge 评分** | 用 GPT-4o 对每条论点的说服力打分 1-10 | 策略二评分更高 |
| **响应时间** | 从提交到完整输出的秒数 | 策略一明显更快 |
| **Token 消耗** | 总 token 数统计 | 策略二消耗更多 |

### 定性评估
- 随机选取 10 个辩题，人工对比两种策略输出
- 重点分析维度：
  - 策略二的论点是否体现了对反方攻击的预判
  - 策略二是否引用了更具体的数据和案例
  - 策略一是否出现明显逻辑漏洞

### 评估流程
```
25个辩题
  → 两种策略各生成3条论点
  → 计算定量指标（自动化）
  → 随机抽10题做人工评估
  → 选2-3个典型案例写入报告
```

## 八、报告结构（对应评分点）

### 1. Motivation（/10）
- 辩论准备的现实需求
- 传统单次生成论点的三个缺陷
- 对抗式生成为何能产生更强论点
- 系统的应用场景和价值

### 2. System Design（/10）
- 整体架构图
- 策略一 vs 策略二的设计对比
- 5个 Agent 的角色设计及理由
- Web Search 集成方式
- 前端实时展示机制

### 3. Dataset（/10）
- 25-30 个辩题的收集来源
- 领域覆盖说明
- 辩题选择标准
- 如何确保辩题质量（有争议、有数据支撑）

### 4. Experiment Design（/10）
- 控制变量说明（相同辩题、相同LLM、相同搜索工具）
- 策略一 vs 策略二的对比框架
- 评估指标选择的理由
- 实验执行流程

### 5. Evaluation Methodology and Results（/10）
- 4个定量指标的结果表格
- 2-3个典型辩题的定性案例分析
- 两种策略的优劣总结
- 适用场景建议（什么时候用策略一，什么时候用策略二）

### 6. Critical Analysis on Ethical Issues（/10）
- **论点偏向性**：AI生成的论点是否存在系统性偏见
- **操纵风险**：强力AI论点工具可能被用于舆论操控
- **信息茧房**：只强化一方立场可能固化用户偏见
- **数据溯源**：Web Search引用内容的真实性无法保证
- **缓解措施**：系统设计层面的应对（展示对方论点、标注信息来源）
